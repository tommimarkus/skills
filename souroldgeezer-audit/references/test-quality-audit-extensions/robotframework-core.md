# Extension: Robot Framework - core

Shared core for the Robot Framework test-quality-audit extension. This file is loaded **whenever Robot Framework test data is detected in the audit target**, before step 0b (rubric selection). It owns detection, test-type dispatch, keyword/library boundary classification, rubric-neutral smells, carve-outs, SUT surface enumeration for keyword-layer audits, determinism verification, and the Robot-level mutation-tool skip declaration.

Rubric-exclusive content lives in the rubric addons:

- [`robotframework-unit.md`](robotframework-unit.md) - keyword-layer unit-test smells and positive signals.
- [`robotframework-integration.md`](robotframework-integration.md) - API, database, CLI, and service contract integration smells.
- [`robotframework-e2e.md`](robotframework-e2e.md) - Browser / SeleniumLibrary / AppiumLibrary E2E smells and positive signals.

Covers `.robot` / `.resource` / `.tsv` suites, resource keywords, Python-backed test libraries, Browser Library, SeleniumLibrary, AppiumLibrary, RequestsLibrary, DatabaseLibrary, Process / SSH / OperatingSystem-driven black-box tests, Pabot-style parallel suites, and Robot's XML / xUnit result artifacts.

Source anchors used for this extension: Robot Framework User Guide (test data, tags, outputs, listener/library model) at https://robotframework.org/robotframework/latest/RobotFrameworkUserGuide.html, Browser Library overview at https://docs.robotframework.org/docs/different_libraries/browser, SeleniumLibrary overview at https://robotframework.org/SeleniumLibrary/, and RequestsLibrary overview at https://docs.robotframework.org/docs/different_libraries/requests.

## Detection signals

Load this extension when the audit target contains any of:

- `.robot`, `.resource`, or `.tsv` files with Robot Framework sections such as `*** Settings ***`, `*** Test Cases ***`, `*** Tasks ***`, `*** Keywords ***`, or `*** Variables ***`.
- `pyproject.toml`, `requirements*.txt`, `Pipfile`, `poetry.lock`, or `uv.lock` containing packages such as `robotframework`, `robotframework-browser`, `robotframework-seleniumlibrary`, `robotframework-requests`, `robotframework-appiumlibrary`, `robotframework-databaselibrary`, `robotframework-sshlibrary`, `robotframework-pabot`, `robotframework-robocop`, or `robotframework-tidy`.
- CI, tox, nox, Makefile, or script commands invoking `robot`, `python -m robot`, `rebot`, or `pabot`.
- Result artifacts under the audit target: `output.xml`, `output.json`, `xunit.xml`, `log.html`, or `report.html` generated by Robot Framework.

Detection glob shortcuts: `**/*.robot`, `**/*.resource`, `**/*.tsv`, `**/requirements*.txt`, `**/pyproject.toml`, `**/output.xml`, `**/xunit*.xml`.

### Runtime flags

Record these on detection; they affect rubric routing and procedure choices:

- **Browser flag.** `Library    Browser`, `robotframework-browser`, or Browser keywords such as `New Page`, `Click`, `Fill Text`, `Get Text`, `Wait For Elements State`, `Get Element States`.
- **Selenium flag.** `Library    SeleniumLibrary`, `Open Browser`, `Click Element`, `Input Text`, `Wait Until Element Is Visible`, `Page Should Contain`.
- **Mobile flag.** `Library    AppiumLibrary` or Appium-style keywords such as `Open Application`, `Click Element`, `Input Text`.
- **API flag.** `Library    RequestsLibrary`, keywords `GET`, `POST`, `Create Session`, `Status Should Be`, `Request Should Be Successful`.
- **Database flag.** `Library    DatabaseLibrary`, keywords `Connect To Database`, `Query`, `Check If Exists In Database`, `Row Count Is`.
- **CLI/process flag.** `Library    Process`, `Library    OperatingSystem`, `Library    SSHLibrary`, keywords `Run Process`, `Start Process`, `Execute Command`, `File Should Exist`.
- **Keyword-layer flag.** Tests target local resource keywords or custom Python libraries directly, and no Browser / Selenium / Appium / API / database / process signal is present.
- **Parallel flag.** `pabot` is the runner or suite tags / commands imply parallel execution.

## Test type detection signals

Consumed by [SKILL.md § 0b (Rubric selection)](../../skills/test-quality-audit/SKILL.md). A Robot Framework test with no matching integration or E2E signal defaults to the unit rubric only when the target is the keyword layer itself; otherwise ask whether the suite is intended as integration or E2E.

### Integration rubric signals - sub-lane A (in-process or adjacent dependency)

Route the test to integration sub-lane A when any of these are present:

- **Project-level.** Directory, suite name, or tag contains `integration`, `contract`, `api`, `database`, `db`, `cli`, or `service`.
- **RequestsLibrary against a local or test-host base URL.** `Create Session` / `GET` / `POST` calls to `${BASE_URL}`, `http://localhost`, `http://127.0.0.1`, docker-compose service names, or test-environment hostnames.
- **DatabaseLibrary against a real database or emulator.** Any `Connect To Database` plus query/assertion keywords, unless the connection is an in-memory fake declared by the suite.
- **Process / SSH / OperatingSystem against the SUT boundary.** `Run Process`, `Start Process`, `Execute Command`, file-system assertions, or SSH commands that exercise the product as a black-box executable or deployed service.
- **Custom library wrapping product APIs.** `Library    <ProductClientLibrary>` plus tests asserting product-level results through that library.

### Integration rubric signals - sub-lane B (out-of-process contract)

Route the test to integration sub-lane B when any of these are present:

- RequestsLibrary calls a deployed environment, shared staging host, external partner sandbox, or contract endpoint outside the local test process.
- Tests assert HTTP status, headers, schema, idempotency, or auth semantics against a published API surface.
- Suite tags include `contract`, `consumer`, `provider`, `staging`, `external`, or `sandbox`.

### E2E rubric signals

Route the test to E2E when any of these are present:

- **Project-level.** Directory, suite, or tags contain `e2e`, `end-to-end`, `ui`, `browser`, `web`, `mobile`, `appium`, `selenium`, or `playwright`.
- **Browser Library.** Browser flag from detection is present.
- **SeleniumLibrary.** Selenium flag from detection is present.
- **AppiumLibrary.** Mobile flag from detection is present.
- **User-session keywords.** Test body includes navigation, real browser / app session setup, click/type/tap gestures, or assertions on rendered page/app state.

Once a test routes to E2E, classify sub-lanes using the core [SKILL.md § 0b step 5](../../skills/test-quality-audit/SKILL.md) plus these Robot-specific signals:

- `[Tags]    a11y` / `accessibility` or axe-style accessibility library usage -> sub-lane **A**.
- `[Tags]    perf` / `performance` or browser performance / timing metrics -> sub-lane **P**.
- `[Tags]    security` / `auth` / `csrf` / `cors` / `cookie` or assertions on session isolation, CSP, CORS, cookie attributes, or tampered tokens -> sub-lane **S**.
- Otherwise -> sub-lane **F**.

### Unit rubric signals (keyword layer)

Route to unit when:

- The test target is a local user keyword, resource file, or custom library keyword, and the test does not hit a real browser, mobile app, API, database, file system outside a temp sandbox, subprocess SUT, or deployed service.
- The suite uses fake or in-memory libraries selected via variables or import arguments, and asserts the keyword's returned value, variable mutation, emitted domain event, or documented keyword output.

### Mixed-file handling

Classify each test case independently. A single `.robot` file can contain keyword-layer unit tests, API integration tests, and Browser E2E tests. The audit records the chosen rubric and, for E2E, sub-lane per test.

## Keyword and library boundary classification

- **User keywords** in `*** Keywords ***` sections and imported `.resource` files are part of the suite's keyword layer. They may be the SUT only when the audit target is the keyword layer itself.
- **Library keywords** from Browser, SeleniumLibrary, AppiumLibrary, RequestsLibrary, DatabaseLibrary, Process, OperatingSystem, SSHLibrary, and custom product libraries are drivers or SUT adapters. They are not the user-facing behavior by themselves.
- **Assertion keywords** include built-in and library keywords whose names contain `Should`, `Must`, `Equals`, `Validate`, `Verify`, `Compare`, `Match`, `Contain`, `Exist`, `Status Should Be`, `Request Should Be Successful`, `Element Should`, `Page Should`, `Wait For Elements State`, or a project-specific assertion wrapper documented in a resource file.
- **Action keywords** include navigation, click/type/tap, process execution, API calls, database writes, and setup/teardown helpers that do not themselves assert a published outcome.
- **Catch/continue keywords** such as `Run Keyword And Ignore Error`, `Run Keyword And Return Status`, `Run Keyword And Continue On Failure`, and broad TRY/EXCEPT blocks are failure-boundary modifiers. Treat them as smells unless the test immediately asserts the captured status or error message.

## Framework-specific high-confidence smells (`robotframework.HC-*`)

These smells apply under unit, integration, and E2E rubrics unless a rubric addon declares a narrower Robot-specific smell.

### `robotframework.HC-1` - Test case has no explicit oracle

**Applies to:** `unit, integration, e2e`

**Detection:** a test case body contains only action/setup keywords and no assertion keyword, no project-specific assertion wrapper, no `Run Keyword And Expect Error`, and no called user keyword whose body contains an assertion keyword. Ignore `[Documentation]`, `[Tags]`, setup, and teardown rows when scanning the body.

**Smell:** Robot's keyword syntax can make a click path or API call read like a scenario, but without an oracle the test only proves that commands did not fail synchronously. The suite is pinning execution, not behavior.

**Example (smell):**
```robotframework
*** Test Cases ***
User Can Open Account Page
    Open Browser    ${BASE_URL}    ${BROWSER}
    Click Element    id=account
    Click Element    id=settings
```

**Rewrite (intent):** assert a user-observable or contract-level outcome after the action path, such as visible page state, response schema, database row, command output, or a keyword return value derived from the requirement.

---

### `robotframework.HC-2` - Failure swallowed by catch/continue keyword without asserting the captured status

**Applies to:** `unit, integration, e2e`

**Detection:** `Run Keyword And Ignore Error`, `Run Keyword And Return Status`, `Run Keyword And Continue On Failure`, or `TRY` / `EXCEPT` around a SUT action where the captured `${status}` / `${error}` is not asserted in the same test.

**Smell:** the test converts a product failure into test control flow, then never checks the result. A failing product path can become a passing test.

**Example (smell):**
```robotframework
${status}    ${message}=    Run Keyword And Ignore Error    Submit Payment
Page Should Contain    Thank you
```

**Rewrite (intent):** use `Run Keyword And Expect Error` for expected failures, or assert both the captured status and the user-visible result before continuing.

---

### `robotframework.HC-3` - Shared suite/global state mutated by a test without reset

**Applies to:** `unit, integration, e2e`

**Detection:** `Set Suite Variable`, `Set Global Variable`, mutable module-level Python library state, or Browser / Selenium sessions shared across tests, with no matching test teardown / suite teardown reset and no per-test resource name.

**Smell:** later tests depend on state left by earlier tests. The suite can pass or fail based on order, parallelism, or rerun selection.

**Rewrite (intent):** keep data local to the test, create per-test resource identifiers, reset mutable library state in teardown, and close browser / API / database sessions per test unless sharing is the explicit behavior under test.

---

### `robotframework.HC-4` - Test template collapses all examples into the same weak assertion

**Applies to:** `unit, integration`

**Detection:** `[Template]` / `Test Template` with several rows where every row shares the same expected literal (`PASS`, `OK`, `200`, `${TRUE}`) and the template body asserts only that generic value or only checks that no error occurred.

**Smell:** the data table looks broad but does not encode case-specific expectations. It is equivalent to one smoke test repeated with different inputs.

**Rewrite (intent):** include case-specific expected values or split into named tests that each prove a distinct equivalence class.

---

## Framework-specific low-confidence smells (`robotframework.LC-*`)

### `robotframework.LC-1` - Branching logic in the test case body

**Applies to:** `unit, integration, e2e`

**Detection:** `IF` / `ELSE`, `FOR`, `WHILE`, or legacy `Run Keyword If` in a `*** Test Cases ***` body, excluding simple data-table templates and setup helpers.

**Why low-confidence:** branching in the test body often means one test encodes multiple behaviors and may skip assertions based on runtime state. It can be legitimate for generated matrix tests or retrying eventually consistent infrastructure.

**Rewrite (intent):** move branching into a named helper keyword whose contract is tested separately, or split the scenario into separate tests with fixed inputs and fixed assertions.

---

### `robotframework.LC-2` - Assertion keyword name hides a selector, endpoint, or implementation detail contract

**Applies to:** `unit, integration, e2e`

**Detection:** a user keyword named like an assertion (`* Should *`, `Verify *`, `Validate *`) whose body asserts only raw CSS/XPath locators, literal endpoint paths, command strings, or database table names, with no business-level expected value.

**Why low-confidence:** wrapper keywords can be a good readability layer. The smell appears when the wrapper only renames an implementation selector as a requirement.

**Rewrite (intent):** keep low-level selectors and paths inside the helper, but make the helper assert a named user-visible state, contract field, or domain outcome.

---

### `robotframework.LC-3` - Runtime-selected library or resource with no audited binding

**Applies to:** `unit, integration, e2e`

**Detection:** `Library    ${LIBRARY}`, `Resource    ${RESOURCE}`, variable-driven environment files, or import arguments selected entirely from environment variables, with no checked-in default or CI matrix explaining which binding is used.

**Why low-confidence:** variable-driven imports can be necessary for cross-environment suites, but they make the audited SUT and runner contract ambiguous.

**Rewrite (intent):** document the binding in CI or argument files, and include the resolved value in audit evidence before applying framework-specific smells.

---

### `robotframework.LC-4` - Template or data-driven rows cover only a narrow happy partition

**Applies to:** `unit, integration, e2e`

**Detection:** a `Test Template`, data-driven table, or repeated keyword call
uses only valid credentials, `200` / `OK` statuses, one valid role, one valid
payload shape, or repeated values from the same equivalence class while the
keyword name, tags, endpoint, schema, or SUT extension exposes invalid,
unauthorized, forbidden, empty, duplicate, boundary, or session-lifecycle
partitions.

**Why low-confidence:** Robot tables can look broad because they have many
rows, but rows that all assert the same happy result are still one narrow
scenario. Generic empty/null/status rows do not count as boundary coverage
unless the SUT contract says those are the relevant edges.

**Rewrite (intent):** add rows for named negative partitions and assert the
observable status, message, page state, or domain outcome for each row.

---

## Framework-specific positive signals (`robotframework.POS-*`)

### `robotframework.POS-1` - Business keyword plus explicit technical oracle

**Applies to:** `unit, integration, e2e`

**Detection:** a test uses a high-level domain keyword for action (`Customer completes checkout`) and follows it with an explicit assertion keyword over visible UI state, API response shape, database row, command output, or returned keyword value.

**Why positive:** this is the Robot style that carries test intent without losing verification detail. The high-level keyword makes the scenario readable; the oracle makes it falsifiable.

---

### `robotframework.POS-2` - Per-test resource isolation

**Applies to:** `unit, integration, e2e`

**Detection:** test data names include `${TEST NAME}` / `${SUITE NAME}` / generated IDs, temp directories, per-test API sessions, per-test browser contexts, or teardown that deletes resources created in the test.

**Why positive:** Robot suites frequently run against shared systems. Per-test resource isolation prevents order dependence and makes `pabot` / selective reruns safer.

---

### `robotframework.POS-3` - Rubric lane declared by tags and confirmed by body

**Applies to:** `unit, integration, e2e`

**Detection:** `[Tags]` or `Test Tags` include lane tags such as `unit`, `integration`, `contract`, `e2e`, `a11y`, `perf`, or `security`, and the test body contains matching technical signals.

**Why positive:** tags are first-class Robot metadata and appear in reports. Matching lane tags make audit dispatch reviewable and reduce false rubric selection.

---

## Carve-outs

- **`Run Keyword And Expect Error` is an oracle.** Do not flag `robotframework.HC-2` when the test uses `Run Keyword And Expect Error` with a specific expected error and asserts the behavior after the failure if state is relevant.
- **Read-only suite variables are configuration.** Do not flag `robotframework.HC-3` for variables set once in suite setup and never mutated by test cases.
- **Template tables are acceptable when every row has a case-specific expected value.** Do not flag `robotframework.HC-4` merely because `[Template]` is present.
- **Generated matrix tests may branch in helpers.** Do not flag `robotframework.LC-1` when branching lives in a tested helper keyword and each generated case still has a fixed oracle.

## SUT surface enumeration

Consumed by [SKILL.md § SUT surface enumeration](../../skills/test-quality-audit/SKILL.md) in deep mode.

Robot Framework itself does not expose the production SUT surface. When another SUT stack extension is detected (.NET, Node.js / TypeScript, Next.js, or future Python), use that extension's SUT surface enumeration for product-code gaps.

Robot tests can still satisfy SUT-stack coverage gaps when they exercise the SUT's public boundary and assert the relevant contract. Example: a RequestsLibrary test against a .NET route can satisfy that route's `.NET` `Gap-Route` entry if it asserts status plus stable body/header/auth/domain behavior. Record this as external contract coverage under the SUT extension. Robot rows that only assert `200`, `OK`, URL reachability, element presence, or successful login are `referenced-weak`; they do not suppress invalid-payload, unauthorized, forbidden, session-lifecycle, duplicate, boundary, or state-change gaps. Do not use Robot evidence to suppress source-level unit seams, private throw-site details, or mutation-target findings that Robot cannot observe.

Use Robot-specific surface enumeration only when the audit scope says the **keyword layer** is the SUT, such as a shared `.resource` library or custom test library intended for reuse.

### Keyword-layer `Gap-API` patterns

- Resource keyword declarations: in `.robot` / `.resource`, inside `*** Keywords ***`, capture non-empty top-level keyword names until the next section header.
- Python library keyword declarations: public functions / methods in files imported by Robot suites, plus `@keyword(...)` decorated functions. Capture the Robot-visible name when the decorator supplies one; otherwise capture the function / method name converted to Robot's space-insensitive keyword style.

### Cross-reference matching

For each enumerated keyword, search audited test cases for a call row whose first cell resolves to that keyword name after Robot's case-insensitive, space-insensitive matching. Suppress keywords tagged or documented as private helpers by naming convention (`_Internal`, `Internal *`, `Helper *`) unless the repo declares them public.

### Confidence annotations

- `Gap-API` for public resource keywords: **medium** - helpers are often intentionally indirect.
- `Gap-API` for public Python library keywords: **medium** unless the library is published as a reusable Robot library, then **high**.

## Determinism verification

Consumed by [SKILL.md § Determinism verification](../../skills/test-quality-audit/SKILL.md) - step 4.5 of the deep-mode workflow. Applies under unit and integration rubrics when the suite is cheap to rerun. Do not run against E2E suites.

### Cheap-rerun command

Run the selected non-E2E Robot target twice with machine-readable output:

```bash
robot --log NONE --report NONE \
  --output ./.test-determinism/run1/output.xml \
  --xunit ./.test-determinism/run1/xunit.xml \
  <robot-target>
robot --log NONE --report NONE \
  --output ./.test-determinism/run2/output.xml \
  --xunit ./.test-determinism/run2/xunit.xml \
  <robot-target>
```

Compare the two xUnit files by test case name and outcome. Any test whose status differs is a runtime-proven flake finding.

### Gating

- **Project size:** skip and recommend targeted reruns when the target has >= 500 test cases. Approximate by counting non-empty rows in `*** Test Cases ***` sections, excluding settings rows and indented keyword rows.
- **Total elapsed time from run 1:** if run 1 takes more than 60 seconds, warn before run 2 and ask before continuing in interactive audits.
- **E2E suites:** never run. Browser / mobile UI suites require target-specific flake investigation instead of a full duplicate run.
- **Parallel suites:** if `pabot` is the normal runner, run determinism in the same parallel mode only if the user explicitly wants to test parallel determinism. Otherwise run serial `robot` to isolate test-order flake from worker scheduling flake.

## Mutation tool

Robot Framework tests usually exercise a product through keywords and libraries; mutation testing belongs to the SUT language, not to Robot test data. When another SUT extension is loaded, use that extension's mutation tool. The Robot extension declares no Robot-level mutation run.

### 1. Tool name and link

No Robot-Framework-level mutation tool is declared. Use the SUT language extension's mutation tool instead, such as Stryker.NET or StrykerJS when those extensions are loaded.

### 2. Install instructions

Do not install a Robot-specific mutation tool for this extension. Install the mutation tool declared by the detected SUT language extension, or skip mutation testing with state B when only Robot Framework is detected.

### 3. Detection command

The audit agent runs this side-effect-free command for Robot-only targets and treats the non-zero exit as "no Robot-level mutation tool":

```bash
false
```

### 4. Run command

No Robot-level run command. If a SUT language extension is loaded, follow that extension's run command against the SUT project, not against `.robot` files.

### 5. Known SUT limitations

- Robot-only audit target: there is no product-code mutation surface. Skip mutation with state B and continue static audit.
- E2E Robot target: mutation testing is not meaningful against a browser/mobile black-box suite. Skip per the core E2E rule.
- Mixed Robot + SUT-language target: run mutation only through the SUT extension and reconcile results with Robot static findings when a Robot test suite is the only executable coverage for that SUT surface.

### 6. Output parser notes

No Robot-level output exists. If a SUT mutation tool ran, parse its report using the SUT extension. Robot `output.xml` / `xunit.xml` files are useful for runtime distribution and determinism, not mutation scoring.
