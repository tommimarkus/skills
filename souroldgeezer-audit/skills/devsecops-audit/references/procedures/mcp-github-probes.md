# MCP GitHub Probes

Deep-mode step 10 of the skill. Conditional on MCP availability. Fetches live GitHub state and compares it to declarative config in the repo.

Rubric: devsecops.md §2.5 (continuously verified), §8 (what static audits cannot know).

## Availability check

At deep-mode step 0c, attempt exactly one MCP call:

```
mcp__github__get_me
```

On success → MCP available. On any error (tool not registered, auth failure, network error, rate limit) → MCP unavailable; skip this entire procedure and emit the single footer disclosure line:

```
MCP github: unavailable — enforcement verification skipped (static findings only).
```

**Never retry.** MCP availability is an environmental property, not a repo property.

## Probe list

When MCP is available, run these probes in order. Each is independent; a single probe failure does not stop later probes.

### Probe 1 — Branch protection on default branch

Call:
```
mcp__github__list_branches (protected=true)
```
And for the default branch specifically, check the returned `protection` field. Expected properties:

- `required_pull_request_reviews.required_approving_review_count` ≥ 1
- `required_pull_request_reviews.dismiss_stale_reviews` = true
- `required_pull_request_reviews.require_code_owner_reviews` = true
- `required_status_checks.strict` = true
- `required_status_checks.contexts` = non-empty list including the CI workflow(s)
- `enforce_admins` = true (no admin bypass)
- `allow_force_pushes` = false

Any missing / false property → `DSO-HC-4` or `CICD-SEC-1` finding with specific property pointer. All passing → `DSO-POS-8` positive signal.

### Probe 2 — Recent workflow runs

Call:
```
mcp__github__actions_list (per_page=30)
```
For the last 30 runs, inspect:

- Pass/fail ratio.
- Whether any run had a security-scan step that was skipped or had `continue-on-error: true` (confirm against the workflow file for `gha.HC-5`).
- Whether deploys ran on PRs (unexpected; investigate).

No universal positive/negative; findings depend on pattern. Report the summary.

### Probe 3 — Dependabot alerts

Call:
```
mcp__github__list_dependabot_alerts (state=open)
```
Count by severity: critical / high / moderate / low. Age of oldest critical / high alert.

Findings:
- Any open critical > 7 days → `DSO-HC-15` (no remediation SLA or SLA not honored).
- Any open high > 30 days → `DSO-HC-15` warn.
- All alerts within SLA → `DSO-POS-7`.

### Probe 4 — Code scanning alerts

Call:
```
mcp__github__list_code_scanning_alerts (state=open)
```
Same pattern as Probe 3: count by severity, age of oldest critical/high.

### Probe 5 — Secret scanning alerts

Call:
```
mcp__github__list_secret_scanning_alerts (state=open)
```
Any open alert → `DSO-HC-1` finding with the specific secret type. Note that GitHub auto-masks the secret value; the finding just cites "secret type X found on commit Y."

### Probe 6 — Collaborator permissions

Call:
```
mcp__github__get_teams
mcp__github__get_team_members (per team)
```
Look for:
- Admin users outside of a well-known ops team.
- Users with write access who have not committed in > 180 days (stale accounts — `CICD-SEC-2`).

### Probe 7 — Workflow permissions baseline

For each workflow file in `.github/workflows/`, cross-reference the declared `permissions:` block against what the workflow actually uses (e.g. if a workflow declares `packages: write` but never pushes a package, that's over-permission). This is a heuristic check; low confidence.

## Output

A `live-state` section appended to the deep-mode report:

```
Live state (MCP github):
  Branch protection (main): enforcing / partial / missing
    - required reviews: N
    - CODEOWNERS required: yes / no
    - admin bypass: disabled / enabled
  Recent workflow runs (last 30): <pass_rate>%
  Open Dependabot alerts: C critical / H high / M moderate (oldest critical: N days)
  Open Code-scanning alerts: ...
  Open Secret-scanning alerts: ...
  Stale collaborators: N users > 180 days inactive with write access
```

## Gotchas

- MCP calls can fail mid-procedure (rate limit, transient error). Continue with the remaining probes; mark the failed probe as "skipped — transient error" in the output.
- Every finding generated by this procedure cites `live-verified` in its severity rubric so the reader can distinguish statically-inferred from runtime-verified findings.
- Never retry an MCP call. One shot per probe.
- Never call `mcp__github__` methods that write state (create branch, push files, merge PR, etc.). Read-only.
